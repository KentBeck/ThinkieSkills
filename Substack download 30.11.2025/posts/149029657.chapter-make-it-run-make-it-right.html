The first day I started programming in BASIC on a PDP-8E (this is probably 1975) I remember my Pappy (himself a programmer) telling me, "Son, make it run, make it right, make it fast." I had no idea what he meant by that but I remembered the phrase.

It was a while before I had a data set big enough that response was less than instantaneous. I looked at my code & realized I had no idea how to make it faster. That’s when Pappy introduced me to tidying. Once it was tidy & profiled, I could see what to fix.

Once I got some confidence, I tried to solve all 3 problems at once:

  • Tidy code

  • That worked correctly

  • And efficiently

I soon discovered that my limited brain could not solve all 3 problems at once. Any time I tried, I ended up with:

  • Messy code

  • That broke unpredictably

  • And was slow

  • And I couldn’t fix any of the 3 problems

Time to throw the whole mess away & start over, by making it run, then making it right, then making it fast.

One Thing At A Time

The paradox of "make it run, make it right, make it fast" is that by solving these 3 problems in sequence instead of in parallel, you get better results on all 3 dimensions. For me, this has to do with cognitive load.

If I’m trying to solve all 3 problems at once I’m just not smart enough to think of & handle all the corner cases, all the combinations of potential problematic input. The same is true of design. I can’t imagine the whole design space & the paths through the design space if I’m worried about correct behavior & efficiency. Similarly, the optimization decisions I make in advance of actual experience running the code on realistic inputs are going to lack focus. Any additional complexity I introduce in the name of efficiency will interfere with my ability to write code I can count on to work. That additional complexity also interferes with my design decisions.

Genius programmers working in a limited scope are able to address all 3 problems at once. I’m not interested in teaching genius programmers new tricks. They’ll learn on their own. It’s folks like me with ordinary programming talent that need help.

If we acccept that solving one out of run/right/fast at a time is more effective, why is run then right then fast the optimal sequence for most programming?

Making code right, that is getting the expected outputs from the expected inputs, is the foundation. All the design transformations we described in the first section, the refactorings, all intend to preserve the inputs → outputs relationship. Before you can apply a semantics-preserving transformation, you need some semantics. The same is true of efficiency. Optimization (outside the scope of these books but very much inside the scope of my practice) operates as semantics- preserving transformations, either doing less work to get to the same answer or less work to get to a similar & good enough answer. If I don’t have to compute the right answer, I can make the program really fast.

That’s it. Really fast. Haven’t sacrificed (non-existent) correctness.

Code that computes correctly can be deployed, so from a net present value perspective we have created value by earning sooner. Maybe it’s untidy & thus hard to change, but this whole book series is about how to handle that situation.

Code that computes correctly is also an emotional boost. Maybe we’re solving a big problem & doubt about our ability to tackle it all is sapping our energy. Hey, it may not be great, but at least it works! Now we have the confidence to take the next step.

Code that computes correctly helps define our design target. We might go into a change thinking great big thoughts about potential designs, but if the code turns out to be simple, we can rapidly trim those thoughts to fit the actual design problem. Untidy but running code gives us a concrete target for our design. The more code we have running, the more accurately we can bound our design thoughts.

The better the behavior of the code, the better our ability to design.

Figure 1. Better running leads to better design

In a tidy first world, we’ve also seen that better design can also enhance our ability to make the next behavioral change. That’s why run/right/fast is not an excuse to go write days or weeks of logic while ignoring or minimizing design. Always consider the possibility that the tidy-first workflow will decrease immediate effort.

Figure 2. Better design leads to easier coding

Finally, tidy code that works provides you with leverage when you go to improve efficiency. Running correctly is an absolute prerequisite & good design adds leverage to your changes. If you have a highly cohesive design, chances are that computation is concentrated in one of those cohesive elements & that changes will be isolated to that element.

Figure 3. Correctness & good design accelerate optimization

Limitations

Highly coupled systems, those that just can’t escape the coupling, aren’t happy homes for the run/right/fast sequence. Efficiency can be correctness. If you slow down a service & that causes other services to slow down & the failures ripple, then you have to pay attention to efficiency at every step through the design & implementation spaces. You can’t afford to make things worse on the way to making them better.

This scenario doesn’t invalidate run/right/fast in general because most programming situations aren’t that constrained. Such a highly constrained system will inevitably evolve more slowly. That may be okay because it’s already big & (one hopes) profitable.
